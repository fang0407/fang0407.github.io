RTCP协议

Posted on 2023-09-23 Edited on 2024-05-04 In Protocol

RTCP介绍

RTCP（Real-time Transport Control Protocol）是用于控制和监视实时多媒体会话的协议。它是 RTP（Real-time Transport Protocol）的伴随协议，通常与 RTP 一起用于音频和视频会话。RTCP 的主要目的是提供会话质量反馈、参与者识别、会话信息维护以及带宽管理等功能。

RTCP包模式

Compound

  • 在 RFC4585 中定义
  • Compound 包至少包含一个 SR 或者 RR 包，随后是至少包含 CNAME 信息的 SDES 包。它的缺点就是 RTCP 包比较大。

Reduced-Size

  • 在 RFC5506 中定义
  • Reduced-Size 包只包含当前 RTCP 类型的必要内容，不会包含多余的 SR 或者 RR 包。比如 NACK 包，就只包含 NACK 相关的信息。这种模式最大的好处就是 RTCP 包的大小大幅减少，有利于网络传输。

在 RFC3550 规范中，发送 RTCP 包强制使用 Compound 包模式。

判断RTP/RTCP包

如何判断一个包是 RTCP 包

  • 长度满足最低的要求，大于 4 字节固定头部
  • 版本信息符合要求，当前 version=2
  • 按照 RFC5761 规范，RTCP 包的 PT 值的后 7 位要在 [64, 96) 范围之内

如何判断一个包是 RTP 包

  • 长度满足最低要求，大于 12 字节固定头部
  • 版本信息符合要求，当前 version=2
  • 按照 RFC5761 规范，PT 值为非预留给 RTCP 包的

RTCP报文格式

报文类型

PT 分组类型 描述
200 SR（Sender report） 发送方报告
201 RR（Receiver report） 接收方报告
202 SDES（Source description） 源描述报告
203 BYE（Goodbye） 离开会话
204 APP（Application-defined） 应用定义

RTCP 固定头部：

  • Version（V）：2bits，协议版本。通常设置为 2。
  • Padding（P）：1bit，是否包含填充信息，如果是 1，表示包含填充信息，其中数据包的最后一个字节表示填充的长度（包含本身这 1个字节）。
  • RC/FMT：5bits，不同的 RTCP 类型，它的含义是不同的，如果是 SR 或者 RR 包，代表的是 Report Block 的个数，可以是 0；如果是 Rtp Feedback 或者 Payload-specific Feedback，代表的是不同的格式。
  • PayloadType：8bits，指定 RTCP 消息的类型。
  • Length：16bits，表示 RTCP 包的长度（不包含固定头部四字节），单位是 32bit。

SR报文格式

RTCP SR 报文是发送端发送给接收端的一种报文，用于提供有关发送端发送的 RTP 数据流的统计信息。

  • SSRC of Packet Sender：发送 SR 数据包的同步信源标识符（SSRC）。
  • NTP Timestamp：以 64 位形式表示的 NTP（Network Time Protocol）时间戳，分为高 32 位和低 32 位；该时间戳的高 32 bit 以NTP 格式表示，从 1900 年 1 月 1 日开始计数，以秒为单位。低 32 bit 表示秒后面的小数。
  • RTP Timestamp：RTP 时间戳（以 RTP 媒体流的时钟为单位），用于同步 RTP 数据包的时间。
  • Sender's Packet Count：发送方发送的 RTP 数据包数量。
  • Sender's Octet Count：发送方发送的字节数（octets），如果发送者改变了 SSRC 会被重置。

RR报文格式

RTCP RR（Receiver Report）报文是实时控制协议（RTCP）中的一种报文类型，用于接收端向发送端发送关于媒体流接收情况的反馈信息。RR 报文通常用于提供关于接收端的统计信息，以便发送端根据这些信息来进行调整以提高数据传输的质量和性能。

  • RTCP report block：在 RR 包中，report block 可能会存在多个，最多可以有 31 个。

  • SSRC：4 字节，表示被反馈的同步信源标识符。

  • Fraction lost：1 字节，上一次发送 RR 包到现在的这一段时间的丢包率，它是一个量化值，范围是 [0, 255]，值越大表示丢包率越高，0 表示没有丢包，127 表示丢包 50%，255 表示丢包 100%。

  • Cumulative number of packets lost：3 字节，表示会话开始后的累计丢包数。

  • Extended highest sequence number received：4 字节，其中低 16 位，表示当前收到的 RTP 包最大的序列号，因为序列号是用 16 位无符号整数表示的，经过一段时间之后，序列号就超出了最大值的范围，重新从 1 开始循环，那么这个字段的高 16 位就表示了当前序列号发生循环的次数。

  • Interarrival jitter：4 字节，包达到时间的抖动。

  • Last SR：4 字节，上一次收到的 SR 包里的 rtp_timestamp，代表了发送端发送 SR 包的时间戳，如果还没有收到任何 SR 包，该值可以设置为 0。

  • Delay since last SR：4 字节，表示上一次收到 SR 包，到现在发送 RR 包，在接收端的延迟时间。

XR报文格式

rrtr(receiver reference time report block)

dlrr(delay since last receiver report)

  • rttr 包中包含接收端发送 XR 包的时间 send_time_ntp。
  • dlrr 包中包含两个时间：
    • last_rr，其实就是接收端的 send_time_ntp
    • delay_since_last_rr，表示上一次收到 rrtr 包，到发送 dlrr 包，在发送端的延迟时间
  • 当接收端收到 XR（dlrr）信息时，就可以计算出 RTT 了，RTT = receive_time_ntp - last_rr - delay_since_last_rr。

NACK报文格式

RTCP NACK（Negative Acknowledgement）是实时控制协议（RTCP）中的一种反馈机制，用于表示接收端未能成功接收到某些数据包的通知。NACK 可以帮助发送端快速识别和重传丢失的数据包，以提高数据传输的可靠性和质量。

RFC4585 规范，关于 QoS 的反馈信息有两种；

  • 传输层的 feedback message（Rtpfb）：Nack、TransportFeedback
  • 特定负载的 feedback message（Psfb）：Pli
  • Feedback Control Information：4 字节，其中前 2 个字节表示丢包的起始参考值，后两个字节 BLP（Bitmap of Lost packets）表示丢包的位图，每个位置表示一个 packet id，如果是 1 表示该位置有丢包，如果是 0，则表示该位置没有丢包。一个 BLP 可以表示 16 个位置，位置的顺序是从低位到高位，例如：

PLI报文格式

PLI（Picture Loss Indication），当发生大量丢包或者其他原因，导致视频帧无法正常解码时，接收端可以发送一个 PLI 包，请求发送端立即解码一个新的关键帧。

PT：206，FMT：1，决定 PLI 报文。

Transport Feedback报文格式

Transport Feedback 报文是用于提供有关实时传输协议（RTP）数据包传输质量的反馈信息的一种 RTCP（Real-Time Control Protocol）报文类型。它提供了接收端对发送端发送的 RTP 数据包的接收情况和传输质量的详细反馈。

  • Base sequence number：2 字节，该 RTCP 包记录的第一个 RTP 包的 seq_no，注意，在反馈的各个 Transport Feedback RTCP 包中，该 seq_no 并不一定是递增的，因为存在包乱序，该 seq_no 可能比之前的 Transport Feedback 中的 seq_no 要小。

  • Packet status count：2 字节，表示该包记录了多少个 RTP 包的信息。

  • Reference time：3 字节，参考时间，RTP 包的到达时间的基准时间，单位为 64ms。

  • Fb pkt.count：1 字节，Transport Feedback 包的序列号，用于检测 Transport Feedback 包的丢包情况。

  • Packet chunk：2 字节，记录 RTP 包的到达状态，这里面保存的是压缩数据，有两种类型的压缩数据块， run length chunk（行程长度编码数据块）和 status vector chunk（状态矢量编码数据块）。

  • Recv delta：可选的，1 字节或者 2 字节，记录 RTP 包到达时间与上一个 RTP 包达到时间的间隔，时间单位为 250us（0.25ms），对于第一个 RTP 包，记录的是与参考时间 reference time 的时间间隔。

    • 如果没有收到数据包，就没有 recv_delta 信息。
    • 如果在 packet chunk 当中记录了一个“数据包收到，时间间隔小”状态的数据包，则 recv delta 使用 1 字节无符号的整数记录时间间隔，范围 [0, 63.75] ms。
    • 如果在 packet chunk 当中记录了一个“数据包收到，时间间隔比较大”状态的数据包，则 recv delta 使用 2 个字节的有符号整数记录时间间隔，范围 [-8192.0 ~ 8191.75] ms。
    • 如果时间间隔超过了范围，则新创建一个 Transport Feedback 数据包。

行程长度编码数据块

  • T：1 bit，如果是 0，表示的是 run length chunk。
  • S：2 bit，表示数据包的状态，00：数据包没有收到；01：数据包收到了，时间间隔小；10：数据包收到了，时间间隔较大或者是负增量；11：保留使用。
  • Run length：13 bit，行程长度，表示处于 S 状态下，连续包的个数，其中 RTP 的序列号是基于 base sequence number 来计算。

示例：

S 为 00，表示数据包没有收到，Run length 为 11011101=221，表示连续 221 个 RTP 包没有收到。

状态矢量编码数据块

  • T：1 bit，如果是 1，表示的是 status vector chunk。

  • S：1 bit，如果是 0，表示用 1 位表示数据包的状态，只有数据包收到（1）和数据包没有收到（0）两种状态；如果是 1，表示用 2 位表示数据包的状态，可以表示 4 种数据包的状态，00：数据包没有收到；01：数据包收到了，时间间隔小；10：数据包收到了，时间间隔较大或者是负增量，11：保留使用。

  • Symbol list，如果 S=0，可以表示 14 个 RTP 包的状态，如果是 S=1，可以表示 7 个包的状态。
