RTP头部扩展

要实现基于延迟的带宽估计，第一步需要计算包组之间的到达延迟差，而新版本的 GCC 将基于延迟的带宽估计移到了发送端进行实现，但是不像接收端，发送端由于不接收 RTP 数据包，无法直接进行包组间到达延迟差的计算。

为了解决这一问题，RTP 注册 transport-cc 头部扩展，记录 transport_sequence_number ，同一个 PeerConnection 连接下的所有流的 transport_sequence_number ，使用统一的计数器进行计数，方便进行同一个 PeerConnection 下的带宽估计。

RTPExtensionType：定义 RTP 扩展类型的枚举。

RtpHeaderExtensionMap：管理 RTP 扩展，内部使用数组管理，数组的索引代表 RTPExtensionType 枚举的值，数组的值代表扩展的 ID。

  • bool RegisterUri(int id, absl::string_view uri)：注册头部扩展

TransportSequenceNumber：transport-cc 具体定义，提供写入和解析 transport_sequence_number 的接口。

Feedback解析

当推流端发送RTP 包时，如果在头部添加了 transport sequence number 扩展，拉流端收到RTP包时会处理这个头部扩展，并向推流端发送transport feedback RTCP包，该包记录了 RTP 包到达接收端的时间信息，以及 RTP 包的丢失信息。发送方基于这些信息来估计延迟梯度。

在标准的 WebRTC 环境下，需要在 SDP 当中添加 RTP 头部扩展描述信息：

  • a=extmap:1 http://www.ietf.org/id/draft-holmer-rmcat-transport-wide-cc-extensions-01
  • id 信息需要与代码中注册的 id保持一致，否则无法处理

TransportFeedback：解析 TransportFeedback RTCP 包，获取接收端收到每个 RTP 包的时间。

拥塞控制模块

TransportFeedbackObserver：观察者模式获得 Feedback 数据。

RtpTransportControllerSend：RTP 传输控制，有独立的线程分别处理 Pacer 和 GCC。

  • OnNetworkOk()：ICE 连接状态反馈，成功连接会创建 GoogCcNetworkController

  • OnAddPacket()：调用 TransportFeedbackAdapter 接口 AddPacket()

  • OnTransportFeedback()：调用 TransportFeedbackAdapter 接口 ProcessTransportFeedback()

  • OnSentPacket()：调用 TransportFeedbackAdapter 接口 ProcessSentPacket()

TransportFeedbackAdapter：处理 TransportFeedback 转换成 webrtc::TransportPacketsFeedback

  • AddPacket()：存储时间窗口内的发送数据包。

  • ProcessSentPacket()：发送 RTP 数据包时调用，更新存储的发送包中发送时间。

  • ProcessTransportFeedback()： TransportFeedback 转换成 webrtc::TransportPacketsFeedback

GoogCcNetworkController：google 拥塞控制模块。

  • OnTransportPacketsFeedbackO()：接受 Feedback 数据，进行估计目标码率。

包组延迟差

InterArrivalDelta：包组时间差计算类。

  • SendTimeGroup current_timestamp_group_：当前到达的包组。
  • SendTimeGroup pre_timestamp_group_：上一个到达的包组。
  • ComputeDeltas()
    • 通过前面的内容，可以获得如下数据
      • 每个包的发送时间
      • 每个包达到对端的时间（换算成发送端的时间，包间隔保持不变）
      • 每个包的大小
      • Transport Feedback 包到达发送端的时间
    • 利用这些数据，可以计算如下数据
      • 两个包之间的发送时间差
      • 两个包之间的到达时间差
      • 两个包之间的数据大小差
    • 计算出这些差值数据之后，可以用计算出这些差值数据之后，可以用于下一步来判断网络是否拥塞。
    • 如果以单个包来计算时间差，会存在比较大的误差，一个重要的原因是：WebRTC 发送数据通常使用pacer，它在静态模式下，通常每隔 5ms 会集中发送一批数据，这些突发数据由于是集中发送，接收的延迟基本相同，接收的时间差值非常小。这些差值数据显然不能用于网络拥塞的判断。
    • 为了避免这种误差，我们会对数据包进行分组，比如 5ms之内的数据分成一组，然后计算包组之间的时间差。
